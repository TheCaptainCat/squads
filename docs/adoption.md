# Adopting squads in an existing project

This guide covers bringing an existing project under `sq` management — whether it already has a
squads-shaped folder, or just a pile of issue/spec markdown you want to migrate while **preserving
the original dates**.

The key tool for history is the global **`--at <when>`** option: every time-based command
(`create`, `status`, `update`, `body`, `comment`, `add-story`/`add-subtask`, `ref add`, …) records
its timestamps from `--at` instead of the wall clock. A human or an LLM driving the migration can
therefore forge the historical dates so the imported history looks real.

```bash
sq --at 2024-01-15            create task "Fix login"      # created_at = that date
sq --at 2024-01-15T09:30:00Z  task 2 status InProgress
```
`--at` accepts an ISO-8601 date or datetime (a bare date is midnight; naïve values are UTC). It is
a **global** option, so it goes *before* the subcommand and applies to that one invocation.

---

## 1. Install and scaffold

```bash
uv tool install squads          # or: uvx --from squads sq …
cd your-project
sq adopt                        # non-destructive: see below
```

**`sq adopt`** is the idempotent cousin of `sq init`:

- creates `.squads.toml` if missing (keeps it otherwise);
- ensures the squad folder + per-type subfolders and the `.claude/` scaffolding;
- **imports** any squads-native `.md` files already present (rebuilds the index + the global
  counter from their frontmatter — same engine as `sq repair`);
- activates the bundled roles it doesn't already have.

Run it as often as you like; it never clobbers authored content. Use `--squad-dir`, `--roles`,
`--no-claude` exactly like `init`.

If your existing files **already** follow the squads layout (type subfolders, `PREFIX-NNNNNN-*.md`
filenames, sq frontmatter), `sq adopt` is all you need — it indexes them in place. If not, convert
them (next section).

### Pre-existing CLAUDE.md and .claude files

If your project already has a hand-written `CLAUDE.md` or `.claude/` directory, `sq adopt` (or
`sq init`) handles the collision non-destructively:

**The managed section is inserted at the top of CLAUDE.md.** If `CLAUDE.md` already exists
with hand-written content, the squads-managed operating-model section (the `<!-- squads:start
-->…<!-- squads:end -->` block) is prepended, and the pre-existing prose survives intact below
it. You'll see a warning:

```
warning: CLAUDE.md had pre-existing hand-written content with no squads markers;
the managed section was inserted at the top — review it for possible contradiction
with that content.
```

The managed section is regenerated by every `sq sync`, so any hand-edits inside its markers are
lost on the next sync. To preserve your operating-model guidance:
- **Move hand-written prose outside the markers** (below the `<!-- squads:end -->` line), or
- **Fold it into the managed section** before the next sync (edit the template instead of the
  rendered file).

Read both blocks to ensure they agree — if your old CLAUDE.md delegates differently than the
squads-generated one, adjust your workflow or override the templates (see `docs/overrides.md`).

**Orphan .claude files are listed as warnings, never deleted.** Pre-existing `.claude/agents/*.md`
pointer files or `.claude/skills/<name>/` directories that don't match any activated role or
custom skill are left in place. You'll see:

```
warning: candidate orphan (not managed by this squad, never auto-deleted): .claude/agents/some-old-role.md
```

This means squads found a `.claude/` file it doesn't generate — perhaps from a previous manual
setup or a custom role you've since renamed. To clean it up:
1. **Verify it's safe.** Check the file to confirm it's not something you want to keep.
2. **Back it up if needed** (before running `sq sync` again).
3. **Remove it by hand** (`rm .claude/agents/some-old-role.md` or `rm -rf .claude/skills/old-skill/`).

Squads never touches these files — you are always in control.

## 2. Convert legacy documents (preserving history, one-by-one)

For each legacy artifact (an old ticket, spec, ADR, …):

1. **Recreate it as a squads item at its original date.** `create` prints the new file path:
   ```bash
   sq --at 2024-01-15 create feature "User authentication" --parent EPIC-1
   # → created FEAT-7 → squads/features/FEAT-7-user-authentication.md
   ```
2. **Move the legacy content into the body** with `sq <type> <n> body` (`--file` is easiest for
   existing prose); for features, scaffold the user stories; for tasks, the subtasks:
   ```bash
   sq --at 2024-01-15 feature 7 body --file legacy-feature.md
   sq --at 2024-01-15 feature 7 add-story "As a user, I want to log in" -m "Acceptance: …"
   sq --at 2024-01-16 task 8 add-subtask "Validate token" --story US1
   ```
3. **Replay the status history** in order, each at its real date:
   ```bash
   sq --at 2024-01-16 task 8 status InProgress
   sq --at 2024-01-20 task 8 status Done
   ```
   (Only the final status persists in the index; the dated discussion entries below are what give
   you the timeline.)
4. **Re-create comments / hand-offs** as dated discussion entries, attributed to the right agent:
   ```bash
   sq --at 2024-01-17T14:00:00Z task 8 comment --as reviewer -m "LGTM, ship it"
   ```
5. **Re-link relationships** (parent, bug/review refs):
   ```bash
   sq --at 2024-01-15 task 8 ref add BUG-9 --kind fixes
   ```

Work in chronological order so each `updated_at` reflects the last real activity.

## 3. Bulk import from JSONL

For large migrations or programmatic conversions, use `sq import` to load a JSONL event stream
in one batch. Each line is a JSON object describing one operation (create an item, set its status,
add a comment, …). The tool validates every event's type, status vocabulary, and actor
registration, collecting **all** problems up front — only a fully clean file is applied, in one
transaction.

```bash
sq import history.jsonl                                  # apply events as-is
sq --at 2024-01-15T09:00:00Z import history.jsonl        # set file-level default timestamp
sq import history.jsonl --as manager --dry-run           # validate without writing
```

For a large or complex file, **always use `--dry-run` first** to see the plan:

```bash
sq import history.jsonl --dry-run
# → lists the projected item allocations and any validation errors
```

**Validation failures write nothing.** If the file has any error (bad JSON, unknown status,
missing parent, …), the whole file is rejected with a line-numbered error report. Fix the file
and re-run — there is nothing to recover because nothing was written.

**Recovering from a mid-apply I/O failure.** A rare I/O crash mid-apply can leave some item
files written but the index commit uncommitted, leaving the index out of sync. If an import
command fails partway through:

1. Run `sq repair` immediately:
   ```bash
   sq repair     # reconciles the index to whatever files exist on disk
   ```
2. Inspect what landed. Use `sq tree` or `sq list` to see which items made it:
   ```bash
   sq tree       # view the current state of all items
   ```
3. **Do not re-run the same import file.** This version of squads has no idempotency — a
   re-run allocates fresh IDs and **duplicates every item**. Instead:
   - Compare the file to what landed and manually add the missing items (`sq create`, one at a
     time).
   - Or start fresh by renumbering and skipping events you've already created.

The `--dry-run` option is your guard against an accidental second run — always validate before
applying to a production squad.

## 4. Validate

```bash
sq check        # markers, dangling parents/refs, invalid status, frontmatter↔index drift
sq tree         # eyeball the imported hierarchy
sq repair       # rebuild the index from frontmatter if anything looks off
```

`sq repair` and `sq check` read timestamps straight from the frontmatter, so the dates you forged
with `--at` are preserved across rebuilds.

## Letting an agent do it

After `sq adopt`, the project's `CLAUDE.md` and the `squads` skill teach the agents the workflow.
Point an agent (e.g. the manager or tech-lead) at the legacy docs and ask it to migrate them
file-by-file using the steps above — it already knows the commands, and `--at` lets it stamp each
item with the date it reads from the source document.

## Notes & caveats

- `--at` sets **both** `created_at` and `updated_at` for a `create`; re-run later commands at later
  dates to advance `updated_at`.
- One timestamp per invocation — run a separate `sq … ` command per historical event.
- The global counter is monotonic: IDs reflect *adoption* order, not historical order. The dates
  (not the ID numbers) carry the timeline.
- A task's parent must be a feature; bugs/reviews attach via `--kind fixes|addresses` (see
  `sq workflow`).
