## Context

Claude Code is the first agent harness squads drives, but it was never meant to be the only one. The
design question was whether to write Claude Code's file conventions directly into the core, or to put
an abstraction between the core and any one harness. We chose the abstraction, and we made a second,
related call: the harness-specific files squads writes into `.claude/` are **pointers**, not content.

The reasoning: tying the core to one harness's directory layout would make a second backend a rewrite,
not an addition. And duplicating real role/skill definitions into `.claude/` would create two copies
that drift, and would scatter durable content into a tool-owned tree that squads regenerates. Keeping
the real content in one place (the squad folder) and treating `.claude/` as a regenerated routing
layer keeps a single source for each definition and keeps the core harness-agnostic.

## Decision

**Backends are pluggable behind the `AgentBackend` ABC, and `.claude/` files are thin pointers.** The
ABC defines the surface (`ensure_scaffold`, `write_managed`, `generate_role_pointer`,
`generate_skill_pointer`, `remove_artifacts`); a registry maps names to backends and the Claude Code
package registers itself on import. **Nothing outside a backend reaches into `.claude/`** — the core
goes through the ABC.

Everything the backend writes into `.claude/` routes to real content under the squad folder: a role
pointer imports its `ROLE-*.md`, a skill pointer imports its skill body, and `CLAUDE.md`'s managed
section (between its `squads` start/end markers) is regenerated from the roster and workflow. The real
role definitions and skill bodies live under `squads/agents/`; `.claude/` holds only the pointers plus
tool-owned config (`settings.json`, merged on write and never clobbered).

## Consequences

What this binds today:

- **A second harness is an additive backend**, not a core change: implement the ABC, register it,
  done. No command may special-case Claude Code outside its backend.
- **No durable content lives in `.claude/`.** Role and skill content has exactly one home (the squad
  folder); the `.claude/` files are regenerated by `sq sync` and are safe to delete and rebuild.
- **`settings.json` is merged, never clobbered** — the backend unions permission entries into an
  existing file rather than overwriting the operator's keys.
- **The pointer indirection costs a hop**: reading a role means following the pointer to the real
  definition. That is the deliberate price of single-sourcing the content and keeping the core
  harness-agnostic.
