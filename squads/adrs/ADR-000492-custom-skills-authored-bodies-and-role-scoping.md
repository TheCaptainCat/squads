# Custom skills: authored bodies and role scoping

## Context

`sq skill add NAME` creates a `skill` meta-item plus a `.claude` pointer, but the skill it
produces is second-class in two ways: it cannot carry an authored instructional body, and it
cannot be preloaded by chosen roles. The motivating use case (BUG-490) is a release-process
runbook authored as a real skill and preloaded for `manager`, `devops`, and `tech-writer`.

This decision was scoped from a manager+operator design session. Before settling it, the current
code was read end-to-end; several of the session's premises did not survive contact with the
implementation. The corrected picture below drives the decision, and the divergences are called
out explicitly in their own section so the record shows what changed and why.

## What the code actually does today (ground truth)

- **`sq skill add`** routes through `Service.create(skill, …)`, which renders
  `agents/skill.md.j2` (title, the `--desc` line, an optional when-to-use block, an empty
  `## Instructions` heading) into a real `:body` marker region, then writes the `.claude`
  pointer. So the body file already exists with an editable `sq:body` marker region.
- **`Service.set_body`** (the engine behind `sq <type> <n> body`) rejects every meta type
  except operator: "its body is generated from its fields." The `sq skill` addressed subgroup
  exposes no `body` verb at all. This — not clobbering — is why a custom skill's body cannot be
  authored.
- **`sq skill <n> regen`** calls `backend.generate_skill_entry`, which writes **only the
  pointer**. It never touches any skill body.
- **`sq sync`** regenerates skill bodies only for the bundled trio (`squads`, `greeting`,
  `sq-memory`, via `_write_managed_skill`) and the `sq-<type>` item-type skills (via
  `_write_item_skills`, iterating PLAYBOOK types and custom item types from the spec). It never
  iterates author-defined SKILL items, so an author-defined skill's body is structurally outside
  the regeneration path.
- **No "regenerated by `sq sync`" warning is written into any skill body.** That stamp lives
  only on the `.claude/skills/<slug>/SKILL.md` pointer and inside the CLAUDE.md / AGENTS.md
  managed regions (`_backends/_managed_region.py`). Skill bodies — bundled or custom — carry no
  such warning today.
- **`skills_for_role(slug)`** (in `_interactions`) is a pure function returning the always-on
  skills plus the role's item-type skills. `generate_role_entry` calls it directly to build the
  pointer's `skills:` YAML, and `activate_role`/`add_dev` snapshot its output into the role
  item's `extra.skills`, which the role body renders into its `## Skills` section. So a role's
  preloaded-skills set surfaces in **two** artifacts: the pointer YAML and the role body.
- **`VALID_REF_KINDS`** is a closed frozenset of 8 kinds (`related`, `blocks`, `depends-on`,
  `implements`, `fixes`, `addresses`, `supersedes`, `duplicates`). `sq check` warns on any ref
  whose kind is outside it. `SquadsDB.backrefs(id)` inverts forward refs but is **kind-agnostic**
  (returns every item whose refs point at the target, regardless of kind).
- **`is_system` is derivable.** The full set of template-owned skill slugs is
  `bundled_skill_slugs()` (squads/greeting/sq-memory + `sq-<PLAYBOOK type>`) unioned with
  `custom_skill_slugs(spec)` (`sq-<custom item type>`). Any SKILL whose slug is outside that
  union is author-defined. `add_skill` already rejects a slug that collides with an existing
  skill, so a custom skill can never shadow a system slug.

## Decision

### Pillar 1 — an authored, persisted body

A custom skill's body becomes authored content that the author owns, exactly like an item body:

- Add `sq skill <n> body -m "…" | --file <path>` (with `--append`), engine
  `set_body`. Relax `set_body`'s meta-type guard to admit a SKILL item **when that skill is not
  template-owned** (see Pillar 2). System skills stay rejected — their body is regenerated by
  sync, so authoring it would be silently discarded.
- The body is written into the existing `:body` marker region (marker-safe, invariant #3). No
  new region, template, or storage is introduced.
- Because `sync` and `regen` structurally never regenerate an author-defined skill's body, the
  authored body already survives both, and `repair` never reads or rewrites body regions at all
  (it rebuilds the index from frontmatter only). No "stop regenerating" change is required — the
  regeneration the session assumed does not exist.

The create-time `agents/skill.md.j2` placeholder (`## Instructions`) remains the starting stub
the author replaces via `sq skill body`.

### Pillar 2 — system vs custom, derived not stored

The system/custom distinction is real and load-bearing, but it is a **pure function of the slug
and the active spec**, so it is derived, not persisted:

```
is_system_skill(slug, spec) := slug in (bundled_skill_slugs() ∪ custom_skill_slugs(spec))
```

This single predicate governs the one behavior that actually needs it: the authoring gate above
(reject `sq skill body` on a template-owned skill; allow it on a custom one). It also drives
display legibility — `sq skill <n> show` labels the skill system (template-owned) or custom
(authored) computed on the fly.

The other two behaviors the session assigned to a stored flag need no flag:

- **Body regeneration** is already gated structurally: `sync` only regenerates the bodies of
  skills it itself produces, and it never produces author-defined ones.
- **`sq skill regen`** is already pointer-only for every skill.

Reconciliation with the invariants:

- **#1 (frontmatter is truth).** Nothing new is stored that isn't reconstructible. The authored
  body was always in the `.md`; the index never carries it. `repair` round-trips it by leaving
  it untouched. The classification is derived, not stored — so there is no redundant persisted
  state to drift.
- **#5 (`.claude` files are pointers, generated).** Unchanged. The skill pointer stays generated
  and regenerated for every skill, system or custom.
- **#7 (generated files stamp themselves).** The pointer self-stamps as always. A custom skill's
  **body is not a generated file** — it is authored, exactly like a task body, which also carries
  no stamp — so #7 does not apply to it. Adding the warning to a custom body would be actively
  wrong: that stamp is the signal that in-session edits are disposable, and here they are not.

### Pillar 3 — role scoping via a forward edge, inverted

- Store the scope link as a **forward edge on the skill**: `SKILL.refs += ROLE-n:scopes`, read
  "this skill scopes to that role." This introduces one new ref kind, `scopes`, added to
  `VALID_REF_KINDS`. The vocabulary is closed to prevent ad-hoc/typo kinds, not frozen against
  reviewed extension; `scopes` is a first-class, reviewed addition.
- Make role→skill membership **data-driven**. Keep the pure system-membership
  `skills_for_role(slug)` in `_interactions` (it cannot see the index — layering). Add a
  service-layer resolver that unions system membership with the inverted scope edges — for role
  `R`, take `backrefs(R.id)`, filter to skills carrying `R.id:scopes` (backrefs is kind-agnostic,
  so the resolver filters by kind), map to slugs — deduped, system-first then scoped. Feed the
  resolved list to the backend (via `BackendContext`, the way `skill_paths` already flows), and
  store it into the role item's `extra.skills`.
- **Both artifacts stay in sync.** The resolved list drives both the pointer `skills:` YAML and
  the role body `## Skills` section. On link/unlink, a partial-sync hook recomputes the resolved
  list for the affected role(s) and rewrites that role's pointer and body-region `## Skills`. A
  full `sq sync` recomputes for every role (it already re-renders role pointers and bodies — the
  resolver simply becomes their skills source), so the hook is an optimization, never the only
  path.
- **CLI: dedicated verbs.** `sq skill <n> link-role <slug>` / `sq skill <n> unlink-role <slug>`
  are the supported surface. They write (or remove) the `ROLE-n:scopes` edge **and** run the
  partial-sync hook. Raw `sq skill <n> ref add ROLE-n --kind scopes` would write the same edge but
  skip the hook, leaving the role pointer stale until the next full sync — so the dedicated verb
  is the sanctioned path.

### Schema and migration

No frontmatter **shape** changes: the classification is derived (nothing stamped), and the scope
edge reuses the `ID:kind` ref shape supported since schema 0.2. The one forward-compatibility
hazard is the new `scopes` kind: a pre-0.11 client reading a 0.11 repo would `sq check`-warn
"unknown ref kind 'scopes'" and silently ignore the scoping. That is exactly the case the schema
gate exists for. So bump `SCHEMA_VERSION` 0.10 → 0.11; the root callback then hard-stops an old
client with "run `sq migrate up`" instead of letting it mishandle the new edges.

The migration is therefore a **schema-stamp only** — a no-op runner (touches no files, returns 0);
`run_pending_migrations` runs `repair` and stamps 0.11 as usual. It is fully mechanical with no
manual runbook. There is no "flag all bundled skills as system" step, because there is no stored
flag to set.

### Repair round-trip

A custom skill's authored body is reconstructed by being left alone: `_rebuild_index_from_disk`
reads each file's frontmatter via `Item.from_frontmatter` and never reads or writes the body
region, and the index stores no body. The `scopes` edges live in frontmatter `refs`, so
`from_frontmatter` reads them back and the rebuilt index carries them. This proves invariant #1
holds for the one file type that used to be treated as fully regenerable: the index carries only
frontmatter-derivable state, and the authored prose lives solely in the `.md` body region.

## Edge cases

- **Unlink.** Removing the `scopes` edge and running the hook drops the skill from that role's
  pointer `skills:` and body `## Skills`; other roles are untouched.
- **Scope to `manager` vs global availability.** Every skill under `.claude/skills/` is
  discoverable by the main agent by name regardless of scoping — scoping controls only which role
  *pointers* auto-preload it. Scoping a skill to `manager` makes the manager pointer preload it;
  leaving it unscoped keeps it globally available (invokable by name) but auto-loaded by no role.
- **Custom skill with no role links.** Fully valid: authored, discoverable, preloaded by nobody.
- **Deleting a scoped role.** The skill's `ROLE-n:scopes` edge dangles and `sq check` flags it as
  a dangling ref for free — a direct benefit of routing the link through `refs` rather than a
  bespoke field.
- **Seeding order and SKILL-id allocation.** Author-defined skills are in neither
  `bundled_skill_slugs()` (init seeding) nor `custom_skill_slugs(spec)` (sync seeding); they
  allocate their SKILL id on demand through the normal `create` transaction at `sq skill add`
  time, exactly as today. The deterministic bundled/custom seeding lists are untouched, so no
  existing SKILL id churns.

## Alternatives considered

- **Persist a `system`/`custom` flag on the skill item (the session's linchpin).** Rejected as
  redundant stored state: the classification is a pure function of slug + spec, so persisting it
  duplicates derivable data (the same smell caught on ADR-322's stored prefix) and adds a data
  migration for no gain. The one argument for storing it is provenance permanence — if a future
  version drops a bundled type, an existing repo's now-orphaned `sq-<type>` body would re-derive
  as "custom." That is an orphan to clean up, not a reason to freeze a stale classification;
  dropping a type is already a separate migration event.
- **Store scoping in a dedicated field (`extra.roles = [slug,…]`) instead of `refs`.** Slug-keyed
  and avoids expanding the ref vocabulary, but it stands up a second forward-edge mechanism with
  bespoke inversion — the opposite of invariant #4's intent of one edge mechanism with computed
  inversion — and forfeits free dangling-ref detection. Rejected in favor of the single `refs`
  mechanism.
- **A body-generating template hook per custom skill (project-level template overrides).** That
  is the explicitly-deferred build item and does not solve authoring — it would still generate,
  not let an author own, the body. Out of scope.
- **Add no schema bump (rely on the `scopes` edge being harmless to old clients).** Rejected: an
  old client silently ignoring scoping and warning on every scoped skill is precisely the
  mishandling the schema gate is meant to prevent.

## Consequences

- A small, well-scoped surface: one relaxed guard + one new `body` verb; one new ref kind + two
  link verbs + a resolver and a partial-sync hook; a no-op schema-stamp migration.
- Custom skill bodies become authored artifacts with the same durability guarantees as item
  bodies — marker-safe, index-independent, repair-stable — without any new storage.
- Role scoping rides the existing forward-edge/backrefs machinery, inherits dangling-ref
  detection, and keeps the pointer and role body as the two synchronized surfaces they already
  are.
- The `scopes` ref kind and the resolver must both be taught to every backend uniformly (the
  resolved-skills list flows through `BackendContext`, not a direct `skills_for_role` call in one
  backend), so a second backend (AGENTS.md) stays consistent.
- Deriving rather than storing the classification means `is_system_skill` must be evaluated
  against the *active* spec at every gate; a project that renames/drops a type re-derives cleanly.

## Divergences from the design-session framing

1. **"sync/regen clobber a custom skill's body; sync must stop regenerating it."** Incorrect:
   `regen` is pointer-only and `sync` never iterates author-defined skills. The real barrier is
   the missing write path (`set_body` meta-guard + no `body` verb). No "stop regenerating" change
   is needed.
2. **"A persisted system/custom flag is the linchpin governing three behaviors."** Two of the
   three (body regeneration, `regen`) are already handled structurally; the third (a body header
   stamp) does not exist to govern. The distinction is kept but derived, not stored — eliminating
   the flag and its migration.
3. **"The managed-file header stamp must be omitted from a custom skill."** There is no stamp on
   any skill body to omit; the stamp is on the pointer (which stays generated for all skills) and
   the CLAUDE.md/AGENTS.md regions. A latent, pre-existing inconsistency — generated/system skill
   bodies also carry no self-stamp, arguably an invariant-#7 gap — is noted but left out of scope.
4. **"A migration flags all bundled skills as system."** With the flag derived, there is nothing
   to stamp; the migration is a no-op schema bump whose only job is to hard-stop pre-0.11 clients
   before they meet a `scopes` edge.
5. **"Invert via the existing `SquadsDB.backrefs`."** Retained, with the caveat that `backrefs`
   is kind-agnostic, so the resolver filters the inverted set to `scopes` edges.
