## Goal
The override drift mechanism (FEAT-14 / TASK-89) ships a per-release content-hash manifest of bundled templates at `src/squads/_rendering/templates_manifest.json`, generated by `scripts/gen_template_manifest.py`. Drift detection only works if that manifest is regenerated whenever a bundled template changes.

REV-97 F1 added a guard test (`test_manifest_freshness_all_bundled_templates`) that **reds the build** if the manifest is stale for the current version — so a forgotten regeneration is now caught. What's missing is the **generation step itself wired into the release/build process**, so a maintainer doesn't have to remember to run it by hand.

## Scope
- Wire `python scripts/gen_template_manifest.py` into the release/build flow (the same place `uv build` runs), so the manifest is regenerated/verified as part of cutting a release — before the wheel is built.
- Make the failure mode loud, not silent: if templates changed without a manifest bump, the pipeline must fail (the guard test already does this in `pytest`; ensure the release flow actually runs it / the generator in --check form).
- Consider a `--check` mode for the generator (verify-without-write) suitable for CI, distinct from the write mode used at release time, if that cleanly separates 'CI proves freshness' from 'release writes the bump'.
- Confirm the manifest continues to ship as package data in the wheel (it does today — keep it verified).
- Document the release step wherever release/build steps are documented.

## Acceptance
- The manifest is regenerated/verified automatically as part of the release/build process — no manual hand-run required.
- A changed bundled template without a manifest bump fails the pipeline.
- The manifest ships in the wheel (verified).
- The release procedure documents the step.

- [2026-06-14T20:21:24Z] Hugo Ops:
  - Wired the template-manifest generation into the release/build process. Here is what changed:
  - **gen_template_manifest.py — added `--check` mode.** The script now accepts `--check` as a flag. Write mode (default) behaves exactly as before. Check mode verifies the manifest is current for the running version without writing — exits 0 if fresh, exits 1 with a clear per-template error message if the manifest is missing, stale, or covers different templates. Exit-code contract in the docstring updated to match.
  - **publish.yml — manifest regenerated before `uv build`.** Added `uv sync --frozen` (needed to resolve scripts/), then `python scripts/gen_template_manifest.py` (write mode) before `uv build`. On a release tag push the manifest is guaranteed to be current when the wheel is assembled.
  - **test.yml — manifest freshness is a named lint gate.** Added `python scripts/gen_template_manifest.py --check` as the `manifest freshness` step in the lint job (alongside ruff/pyright). Any PR that changes a bundled template without re-running the generator will fail the lint job by name — complementing the existing `test_manifest_freshness_all_bundled_templates` guard in pytest.
  - **CONTRIBUTING.md — Cutting a release section added.** Documents the four-step release procedure (bump version, add changelog entry, run the generator, commit+tag+push) with the `--check` verification command and a note about CI enforcement. This is the canonical home for build/release steps in this repo.
  - **Gates verified:** 543 passed, 1 skipped (same count). ruff/format/pyright clean. `uv build` produces `squads/_rendering/templates_manifest.json` inside the wheel. `--check` passes on a fresh manifest and exits 1 (with a named stale-template report) on a deliberately corrupted hash.
  - No new CI system introduced — all changes are within the existing GitHub Actions workflows the project already uses. No decisions required.
