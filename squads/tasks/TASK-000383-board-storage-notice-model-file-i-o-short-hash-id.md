Build the storage/model layer for the team bulletin board on the model fixed by the accepted decision (ADR-314): its own lighter store under `squads/board/`, one file per notice, off the global counter and outside `.squads.json`.

## Scope

- **Board-notice model** — a lightweight model of its own: a short-hash stable id, author, posted-at, an optional `until` expiry, and a body. NOT the `Item` model; no status lifecycle, sub-entities, refs, or workflow. No global-counter id.

- **File read/write** — write a notice as its own `.md` under `squads/board/` (short-hash id in frontmatter alongside author/posted-at/until, freeform marker-free body); read notices back; delete one by id.

- **Board `.index.jsonl`** — generated by the SHARED generator built in the memory storage task (same format: header line + `{slug, filename, description}` entry lines), written in the same sorted, unexpired order the CLI lists in. Here `slug` carries the notice's stable short-hash id. Reuse the generator; do not duplicate it.

- **Read-time expiry filter** — expired notices (past `until`) are excluded from the generated index and listings at read time. Reads never mutate git-tracked files; physical removal happens only on explicit `clear`.

## Notes

- The positional ordinal is the entry's line position in the generated index (header excluded); it is a display affordance, not persisted meaning — the stored id is the short hash.

- Two notices posted on separate branches get distinct hash ids and do not collide on merge.

- The board's `.index.jsonl` is committed (same posture as `.squads.json`: a rebuildable derived index committed for convenience) and regenerated whole from the notice `.md` files on post/clear and on `sq sync`/`sq repair`, via the shared generator. On a merge conflict of the committed index the fix is mechanical: regenerate from the `.md` files.
