When `init`/`adopt` inserts the squads-managed region into a `CLAUDE.md` (or `AGENTS.md`) that already holds real hand-written content but no squads markers, insert it anyway — but WARN that the pre-existing operating model may contradict the managed instructions.

## Scope

- Detect the case: the target file exists, has non-whitespace content, and carries NO `squads:start`/`squads:end` markers yet. (An already-managed file — markers present — is the normal regen path and gets no warning.)
- Still insert the managed region (do not refuse) and emit a WARNING that the hand-written content and the managed block may contradict each other, so the adopter reconciles them.
- **Consider leading with the managed block**: place the managed region at the TOP of the file on first insertion into pre-existing content, so the authoritative instructions come first, rather than appending below existing prose. Today `_managed_region.inject` appends the block when no markers are found (`f"{text}{sep}\n{block}"`) — evaluate switching first-insertion-into-existing-content to prepend. If prepend is adopted, keep idempotent regen intact: once markers exist, in-place replacement (the existing `si/ei` splice) must continue to update the block wherever it sits without moving it on every run.
- Keep it backend-neutral: the logic belongs in the shared `_backends/_managed_region.py` wrapper (`inject`/`wrap`) so both the `_claude_code` and `_agents_md` backends inherit it, rather than duplicating in each per-file wrapper (`_claude_code/_claude_md.py`, `_agents_md/_managed.py`). `inject` currently returns nothing — have it (or a thin caller-side signal) report whether it inserted into pre-existing unmanaged content so the run can raise the warning.

## Where it plugs in
- `_backends/_managed_region.py` (`inject`/`wrap` — the shared mechanism, already stamps the "regenerated by `sq sync`" warning).
- `_backends/_claude_code/_claude_md.py` + `_backends/_agents_md/_managed.py` (the per-file callers that supply the missing-header text).
- Surface the warning up through `write_managed` -> `refresh_managed` -> the `init`/`adopt` result for the CLI to print.

## Conventions (must hold)
- No status/lifecycle prose. Category is "roster". No ticket/sq IDs in source or test names. PEP-695 `type` aliases. `SquadsError` family (warning, not a hard error — never refuse the insert). Escape console output via `_cli._common.e()`.
- Marker-safe: the `squads:start`/`squads:end` markers and the tool-generated warning line are owned by `_managed_region` — do not hand-write or duplicate them (invariant #7: the managed region says in-place that it's regenerated by `sq sync`).

- Service-level tests: (a) pre-existing hand-written `CLAUDE.md`, no markers -> block inserted (at top, if prepend is adopted), warning emitted, hand-written content preserved intact; (b) already-managed file -> block replaced in place, no warning, no move; (c) missing file -> created with header + block, no warning. Cover both backends (`CLAUDE.md` and `AGENTS.md`) since the logic is shared. CLI smoke test that the warning prints.
