Adopter-facing runbook for adopting squads into a project that already has a hand-written `CLAUDE.md` and/or `.claude` directory. Depends on the orphan-warning and CLAUDE.md-contradiction behaviours being built (this doc describes what an adopter sees and does).

## Scope

Add a section to `docs/adoption.md` covering the pre-existing-corpus collision case:
- **What a run overwrites** — `.claude` pointer files whose slug matches a squads-generated one are regenerated (replaced). Name the pattern so an adopter knows what to back up first.
- **What it leaves as a candidate orphan** — pre-existing `.claude` pointer/skill files that don't match anything squads generates are left in place and reported as candidate orphans; squads never deletes them. Explain how to read that warning and reconcile by hand.
- **Reconciling a hand-written `CLAUDE.md`** — when the managed region is inserted next to existing prose, the two may state conflicting operating models; the managed block leads so its instructions are authoritative. Show how to fold hand-written guidance into (or alongside) the managed region, and note the managed region is regenerated by `sq sync` so hand-edits inside its markers are lost.

Keep it practical: what to check before adopting, what the warnings mean, the manual steps to reconcile. Cross-link the existing `sq adopt` material already in `docs/adoption.md`.

## Conventions (must hold)
- **Adopter-facing only.** No sq/ticket IDs, no internal dev-process content (CI gates, dogfood, packaging/test internals), no reference to this feature or its stories. This is tool documentation for someone bringing squads into their own project.
- No status/lifecycle prose. Category is "roster" if the word arises. Match the existing voice/structure of `docs/adoption.md`.
