## Problem

The "regenerated by `sq sync`; do not edit by hand" warning is stamped in the wrong place.

Today it lives inside the `sq:body` of the bundled squad-skill templates
(`src/squads/_rendering/templates/agents/{greeting_skill,squads_skill,item_skill}.md.j2`, lines
3-4), which stamp `<!-- squads:managed — regenerated by `sq sync`; do not edit by hand. -->` plus
`<!-- squads:version:X -->` into every generated `squads/agents/skills/SKILL-*.md` file (all 9
skill files today: greeting, squads, sq-bug, sq-decision, sq-epic, sq-feature, sq-guide, sq-review,
sq-task).

This is **redundant**: everything under `squads/**` is already sq-managed data — mutated only
through the `sq` CLI. `CLAUDE.md` already states this once, globally ("the `.md` files are
sq-managed — never edit them by hand"). Repeating a bespoke "do not edit" banner per skill file
adds no information and is one more thing to keep in sync.

Meanwhile the surfaces that are genuinely **regenerated on every `sq sync` and directly at risk of
silent hand-edit loss** carry no such warning where it matters:

- **`CLAUDE.md`'s managed section** (rendered from
  `src/squads/_rendering/templates/claude/claude_section.md.j2` via
  `_backends/_claude_code/_claude_md.py`) has no wrapping marker/warning at all in the rendered
  output.
- **`AGENTS.md`** — not yet written by any backend, but is the natural cross-tool equivalent
  (Copilot/Cursor/other agents read `AGENTS.md` the way Claude Code reads `CLAUDE.md`) and will
  need the same treatment when it lands.
- The **`.claude/` pointer files** (`.claude/agents/*.md`, `.claude/skills/*/*.md`) carry only a
  partial reminder — "Never edit the `<!-- sq:* -->` marker lines in managed files" — which is
  about sq item markers, not about the pointer file itself being fully regenerated by `sq sync`.

An agent (or a human) working directly in `CLAUDE.md`, `AGENTS.md`, or a `.claude/` pointer file
today has no local signal that their edit will be silently clobbered on the next sync. That's
exactly backwards: the redundant warning sits on the file that's already covered by a global rule,
and the real risk sits on files with none.

## The two-audience design

There are two distinct messages for two different audiences, and they've been conflated:

1. **"sq-managed data — mutate via the `sq` CLI, never hand-edit."** Applies to *every* file under
   `squads/**` (items, roles, skills bodies, the index). Already stated once, globally, in
   `CLAUDE.md`. No per-file repetition needed — remove it from the skill-body templates.

2. **"tool-generated — hand-edits here are overwritten by `sq sync`."** Belongs on the
   **agent-facing generated files**: the CLAUDE.md managed section, AGENTS.md (once it exists), and
   every per-backend skill/agent/pointer file. This is the warning that actually protects against
   lost work, because these are the files an agent opens as *working context* and could edit
   in-session without realizing they're derived output.

The second warning is a **per-backend rendering concern**, not backend-agnostic squad data: each
backend stamps its own warning into the agent files *it* generates, the same way each backend
already owns its own pointer format. This fits the existing pluggable-backend model
(`AgentBackend` ABC in `_backends/`) — Claude Code stamps it today; a future Copilot/Cursor/
AGENTS.md-only backend stamps its own instance of the same convention tomorrow.

## Scope

1. **Remove the redundant marker** from the squad-skill body templates
   (`greeting_skill.md.j2`, `squads_skill.md.j2`, `item_skill.md.j2`) — drop the
   `<!-- squads:managed — regenerated by sq sync; do not edit by hand. -->` line. Regenerate the 9
   bundled + any per-repo skill goldens.

2. **Decide the fate of the per-file `<!-- squads:version:X -->` stamp** (see open question below)
   and implement that call consistently across the same three templates.

3. **Add a real "managed by squads — regenerated by `sq sync`; do not edit by hand" marker** inside
   the rendered **managed regions** of:
   - `CLAUDE.md`'s managed section (`claude_section.md.j2` / `_claude_md.py`'s insert-or-replace
     logic) — the marker should sit inside the managed markers so it's visually obvious the whole
     block is generated, not just implied by a comment in this repo's own docs.
   - `AGENTS.md`, when a backend writes one (scope the convention now even if no backend emits
     AGENTS.md yet; at minimum the tests/utility for "does the managed-region wrapper carry the
     warning" should be backend-agnostic so the first AGENTS.md writer gets it for free).

4. **Make the per-backend skill/agent pointer files carry it consistently** — the Claude Code
   `.claude/agents/*.md` and `.claude/skills/*/*.md` pointer files should state plainly that the
   *pointer file itself* is regenerated by `sq sync`, distinct from (and in addition to) the
   existing "never edit the `sq:*` marker lines" reminder, which is about item-body markers inside
   squad files, not about the pointer file's own regeneration.

5. **Establish this as a cross-backend convention.** Document (in the backend ABC's docstring
   and/or a short section of the architecture notes in `CLAUDE.md`) that any `AgentBackend`
   implementation must stamp its own generated agent-facing files with a "tool-generated by sq
   sync" warning — this is a contract new backends must satisfy, not an implementation detail of
   the Claude Code backend.

## Open question — the `squads:version` stamp

The per-file `<!-- squads:version:X -->` stamp duplicates the single `squads_version` value already
recorded in `.squads.toml`, and it churned all 9 skill files on the 0.6.0 version bump for no
behavioral gain (it isn't read back or diffed against anything at drift-detection time as far as
this draft has found).

**My call: drop the per-file stamp in favor of the single `.squads.toml` config value**, unless the
tech lead finds an actual consumer of the per-file stamp (e.g. a repair/drift check that compares a
skill file's stamped version against something) that `.squads.toml` doesn't already serve. If such
a consumer exists, keep it — but then it should earn its keep as a documented invariant, not ride
along as a leftover of the same template lines this feature is otherwise pruning. Flagging this
explicitly for the tech lead to confirm or override during task breakdown.

## Cross-backend convention / possible ADR

`op-pierre` raised whether this warrants a companion ADR. My assessment: this is a **rendering/
copy convention**, not a schema or data-model decision — it doesn't change `SCHEMA_VERSION`, the
frontmatter contract, or anything `sq migrate` needs to know about. It's closer to a documented
contract on the `AgentBackend` ABC (point 5 above) than an ADR-worthy architectural decision.
**My call: no standalone ADR needed** — capture the convention in the ABC docstring/CLAUDE.md
architecture notes as scoped above. If the tech lead disagrees (e.g. because they see it as
extending the FEAT-13 stability contract's tier for "generated `.claude/` files"), an ADR is a
cheap add and they should feel free to open one — but I don't think the decision is contentious or
consequential enough on its own to need one.

## Constraints / guardrails

- This **deliberately changes generated managed output** — CLAUDE.md's managed section, the 9
  bundled skill files, and the `.claude/` pointer files will all change. That's intended, not a
  regression, but the implementer must:
  - regenerate every affected golden (`sq sync` on the bundled fixtures + this repo's own
    `squads/agents/skills/*`, `.claude/**`, `CLAUDE.md`);
  - apply the managed-section diff discipline already learned the hard way: diff on-disk
    `.claude`/CLAUDE.md/AGENTS.md output with the **roster held constant** — a dev-less fresh-init
    compared against a dev-bearing repo produces false-positive "regressions" that are just roster
    differences, not this change.
- This interacts with **FEAT-13** (the stability contract): tier 5 of that contract already
  states generated `.claude/` files are "regenerable, never migrated; deleting them loses nothing."
  This feature doesn't change that tier's promise, but it changes *what the generated files say
  about themselves*, so the two must stay coherent — link `related` and have whoever implements
  this re-read FEAT-13's body before touching the wording.
- No schema/index changes; no migration needed — this is template/rendering-only. Confirm `sq
  check` and the full golden-diff suite stay green (mechanically different content, same
  structural rules).

## Acceptance

- The `<!-- squads:managed — regenerated by sq sync; do not edit by hand. -->` line is removed from
  the three skill-body templates; the fate of `<!-- squads:version:X -->` is decided and
  implemented per the call above (or the tech lead's override, recorded in a comment).
- `CLAUDE.md`'s managed section and (when a backend emits it) `AGENTS.md` carry a clear
  "managed by squads — regenerated by `sq sync`; do not edit by hand" warning inside their managed
  region.
- The Claude Code `.claude/` pointer files (agents + skill pointers) state that the pointer file
  itself is regenerated by `sq sync`, distinct from the existing sq-marker reminder.
- The convention is documented as a cross-backend contract (ABC docstring and/or CLAUDE.md
  architecture notes), not a Claude-Code-only implementation detail.
- All bundled + this-repo goldens are regenerated and the full test/golden-diff suite is green.
