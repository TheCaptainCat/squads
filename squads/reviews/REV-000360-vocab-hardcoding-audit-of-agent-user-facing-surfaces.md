Automated vocab-hardcoding audit (FEAT-336) across five facing surfaces: CLI help/messages, Jinja templates, shipped docs, generated skills/roles, and services-core logic. A defect = a facing surface that assumes the bundled vocabulary instead of deriving it from the active spec, so a squad with custom types/statuses/kinds/badges sees wrong, missing, or misleading output. Reserved meta-types (role/skill/operator) are excluded by design (ADR-266).

Total: %d findings. Several highs are wrong even under the *bundled* spec (not only custom vocab) — flagged inline. Findings on `workflow.md.j2` overlap FEAT-334's scope and are owned there.

## HIGH (5)

- **[Jinja2] src/squads/_rendering/templates/items/review.md.j2:13** — The findings severity legend is fully hardcoded: `_Severity:_ 🔴 critical · 🟠 high · 🟡 medium · 🟢 low · 🔵 info`. Post-FEAT-327 the severity axis is a spec-defined Collection (values, labels, emoji all overridable). This legend cannot reflect any customization — the item-template render context (see _services/_base.py:351-353: `render(self._template_for(item_type), item=item, description=description, extra=item.extra)`) does not even pass the active spec, so a project that renames severity levels, changes emoji, or adds/removes levels gets a stale, wrong legend on every scaffolded review body. This is the exact defect class FEAT-336 targets.
- **[docs/**] docs/overrides.md:1-88 (whole doc; layout tree lines 46-76)** — The dedicated customization guide documents ONLY template overrides and role overrides. It never mentions the workflow-spec override (.overrides/workflow.toml) that defines the item-type, status, lifecycle, and priority/severity-badge vocabulary — grep finds zero occurrences of workflow.toml, [items.*], [statuses.*], or [collections.*]. The 'Override layout' tree (lines 47-76) shows templates/ and roles/ but omits workflow.toml. A reader of the doc whose whole purpose is 'how to customize' concludes the type/status/priority vocabulary is fixed and non-overridable, when post-EPIC-325 it is spec-driven and overridable (verified: src/squads/_workflow/_loader.py merges .overrides/workflow.toml additively for items/statuses/lifecycles/collections).
- **[skills-roles] src/squads/_backends/_claude_code/_backend.py:232 (renders src/squads/_rendering/templates/agents/item_skill.md.j2:5; snapshot e.g. squads/agents/skills/SKILL-000199-sq-task.md)** — Every built-in-type sq-<type> skill renders its `**Lifecycle:**` line from `lifecycle=pb.lifecycle` — the FROZEN prose string in playbook.toml — instead of the active spec's status machine. The parallel custom-type branch (line 251) correctly uses `linearize_lifecycle(ctx.spec.machine_for(ctype))`. So a project that keeps a built-in type name but overrides its statuses (renames/adds/drops a status) gets a stale, wrong lifecycle line in that type's skill, while a custom type gets a correct one. Asymmetric and misleading.
- **[skills-roles] src/squads/_rendering/templates/agents/item_skill.md.j2:53 (built-in branch at _backend.py:226-234 omits `subentity_kind`; snapshots SKILL-000195-sq-epic.md:81, SKILL-000194-sq-decision.md:66, SKILL-000193-sq-bug.md:96, SKILL-000197-sq-guide.md:76)** — The trailer hardcodes the literal triple 'user stories / subtasks / findings' for EVERY item type. The PLAYBOOK (built-in) render call never passes `subentity_kind`, so the `if subentity_kind is not defined ...` gate is always truthy: the clause prints for epic/decision/guide/bug (which have no matching sub-entity at all) and over-lists for feature (story only), task (subtask only), review (finding only). The custom branch correctly passes `subentity_kind=None`.
- **[skills-roles] src/squads/_rendering/templates/agents/squads_skill.md.j2:88 (also line 76; snapshot SKILL-000200-squads.md:234,222)** — 'Set importance with `--priority urgent|high|medium|low`' hardcodes both the priority field name and its badge values. Post-EPIC-325 priority is a spec-defined, overridable badge collection (field label + badge set both customizable). A project that renames the field or changes the values gets guidance that names options that don't exist. Line 76's `--priority high` example carries the same assumption.
