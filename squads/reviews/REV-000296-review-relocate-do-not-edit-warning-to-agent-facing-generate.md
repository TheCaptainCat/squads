### F1 — BackendContext.version now unread — vestigial after removing the skill-render kwargs

<!-- sq:finding:F1:head -->
**Status:** 🟡 Fixed
**Severity:** 🟢 Low
<!-- sq:finding:F1:head:end -->

<!-- sq:finding:F1:body -->
Severity: low (cleanliness / non-blocking — no functional impact).

File: src/squads/_backends/_base.py:44,59,64 (the `version` field, constructor param, and `self.version = version`).

Before this task, `BackendContext.version` had exactly three readers: the `render(..., version=ctx.version)` calls in `_claude_code/_backend.py` that fed the now-removed `{{ version }}` skill-file stamp. This diff removes all three. A grep across src/ and tests/ shows nothing reads a BackendContext's `.version` anymore — it is still a required constructor argument passed `version=__version__` at every call site (`_services/_base.py`, `_services/_maintenance.py`, and the two conformance/agents_md test fixtures) and stored, but never consumed.

Note ST1's stated bar was "removed with no orphaned param" — the render kwargs are clean, but the field they were sourced from is now the orphan. Importantly this is NOT a bug: the canonical `.squads.toml` `squads_version` stamp is written independently via `_maintenance._stamp_version(__version__)`, so nothing user-visible breaks.

Fix options (either is fine; I lean toward keeping it, hence low/non-blocking):
1. Drop the `version` field + constructor param and its four call-site arguments if backends genuinely no longer need the running version — smallest dead-code removal, but touches the ABC context contract and both conformance fixtures.
2. Deliberately keep it: `write_managed` is documented as writing "version-dependent files", and a future backend may legitimately read `ctx.version`. If kept, that intent is worth a one-line comment on the field so it doesn't read as leftover.

Recommend a fast-follow (or a WontFix with rationale) rather than blocking this task — the relocation work itself is complete and correct.
