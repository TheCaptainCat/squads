### F1 — US5.2 violated: two distinct memory adds conflict on the committed .index.jsonl

<!-- sq:finding:F1:head -->
**Status:** 🔴 Open
**Severity:** 🟡 Medium
<!-- sq:finding:F1:head:end -->

<!-- sq:finding:F1:body -->
US5 acceptance point 2 states: 'Two branches each adding a distinct memory merge with no conflict.' Verified empirically — FALSE for the generated index.

The slug-named .md content files DO merge cleanly (independent files, as ADR-314 promises). But every add rewrites the whole per-role .index.jsonl. Two branches each adding a DISTINCT memory both rewrite that one committed file with a different entry line, so a 3-way git merge conflicts on .index.jsonl (reproduced: 'CONFLICT (content): Merge conflict in squads/agents/memory/manager/.index.jsonl').

This also contradicts ADR-314's own claim that the Option-A index is 'never the merge battleground'. Root cause: a derived, whole-rewritten roll-up is committed (not gitignored). Boot-surfacing reads this committed index (read_index) instead of deriving lines from the .md files (list_entries), which is what forces it to be committed. No data is lost (the .md source of truth merges), so severity is medium, not high — but a stated acceptance criterion demonstrably fails.

Options: (a) gitignore .index.jsonl and rebuild it on sync + read fresh from the .md files at boot (aligns with 'don't store what you can derive'); or (b) keep it committed but make the conflict trivially resolvable — see the next finding, the documented resolution path is currently broken.

### F2 — index GENERATED_STAMP promises 'sq sync' regeneration that does not exist

<!-- sq:finding:F2:head -->
**Status:** 🔴 Open
**Severity:** 🟡 Medium
<!-- sq:finding:F2:head:end -->

<!-- sq:finding:F2:body -->
Every generated .index.jsonl carries GENERATED_STAMP = 'regenerated by sq on add/forget/edit and sq sync — do not hand-edit' (_content_index.py). Two parts of that promise are false.

(1) 'sq sync' does NOT regenerate memory indexes. regenerate()/_regenerate_index is called only from _memory/_store.py add() and forget(). MaintenanceMixin.sync() (_services/_maintenance.py) regenerates role/skill entries + managed files and now READS the index via memory_index_lines, but never rebuilds it. So sync's docstring ('Regenerate all tool-owned managed files') does not hold for this tool-owned generated file.

(2) There is no 'edit' path — no sq memory edit command; the store never edits a memory in place. So of 'add/forget/edit and sq sync', only add and forget actually regenerate.

Impact: this violates the invariant #7 spirit (a generated file must truthfully say how it is regenerated so an in-session editor does not lose edits). It also breaks the F1 resolution path: after a merge conflict on the 'do-not-hand-edit' index, the naturally-documented fix (sq sync) does nothing, leaving no non-mutating command to rebuild it — the user must hand-edit (violating the stamp) or run a spurious add/forget. Fix: either wire sync to regenerate every role's memory index, or correct the stamp to name only the paths that actually regenerate (and add a rebuild command if the committed-index design is kept).

### F3 — US5 merge/branch behavior has no test — only the off-counter/repair-neutral point is asserted

<!-- sq:finding:F3:head -->
**Status:** 🔴 Open
**Severity:** 🟢 Low
<!-- sq:finding:F3:head:end -->

<!-- sq:finding:F3:body -->
Coverage gap Pierre specifically flagged. US5 has four acceptance points; only point 4 (off .squads.json / off the counter / sq repair leaves memory alone) is tested (service tests test_memory_add_and_forget_never_allocate_a_counter_id... and test_repair_neither_rebuilds...). Point 1 (committed, not gitignored), point 2 (distinct adds merge with no conflict) and point 3 (same-memory edits surface an honest conflict) have NO test.

Because points 2/3 were never exercised, F1 (the index conflict) slipped through. A git-backed test (init a repo in tmp_path, branch, add-on-each, merge) would have caught it, and would pin point 3's desired-conflict-on-same-.md behavior too. Recommend adding it as part of fixing F1.

Rest of the feature is well-covered: US1 (add/slug/frontmatter/no-counter), US3 (list/search/show + clean errors), US4 (forget real deletion + clean error) have solid service+CLI tests; US2 (boot-surfacing) has good parametrized integration tests over both backends (empty pool → nothing, index-only never body, refreshed via the backend regeneration path). Invariant #6 is genuinely satisfied — surfacing goes through the AgentBackend ABC via the shared _memory_surface formatter, not a hard-coded .claude/ reach.
